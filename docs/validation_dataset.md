# Validation Dataset Build Protocol

This protocol applies within the dependency and ownership rules defined in
`architecture.md`. Validation establishes confidence in production source; it
must not introduce an independent hidden implementation.

This protocol is the standard for adding a new model/product result database.
The current standard is production-first: build the target production database,
then validate a small production slice with all reference engines.

## Registry Tiers

Use `registry/production` for target model, product, and result databases. A
new production result can initially be small, for example 500 or 1000 priced
rows, so the audit cycle is fast. Scaling to larger row counts should not
change the numerical protocol.

Use `registry/validation` for deterministic slices cut from production
databases and for their reference repricing results.

Registry files are grouped by family, while database ids remain unchanged:

```text
registry/<tier>/models/<model_family>/{data,specifications,generators}/...
registry/<tier>/curves/<curve_family>/{data,specifications,generators}/...
registry/<tier>/products/<product_family>/{data,specifications,generators}/...
registry/<tier>/results/<model_family>/<product_family>/{data,specifications,generators}/...
```

The JSON stores data. The YAML documents the database. The generator script has
the same basename as the database.

Every model YAML must contain a compact `dynamics` block between `parameters`
and `construction`. It documents the risk-neutral equations, correlations, and
initial conditions using the JSON parameter names. Numerical schemes, RNGs,
time grids, path counts, and devices do not belong there; they remain result
database metadata.

## Production Slice Audit

Every production result database should have a matching validation audit slice.
The canonical Heston fixed-lookback production audit uses:

- production model database: `heston_01`;
- production product database: `lookback_fixed_calls_01`;
- production result database generated by `cpp_gpu_philox`;
- validation slices containing the first 100 model and product rows.

Each model, curve when present, and product slice must have a generator with the same basename. The
generator reads the production JSON, keeps the selected rows, and writes a
validation JSON/YAML that contains the production source path and slice rule.
Production results are not copied into validation; the notebook reads the
first 100 rows directly from the production result JSON.

The validation audit then regenerates prices on the same slice for:

- `cpp_gpu_philox`;
- `cpp_cpu_philox`;
- `python_gpu_pytorch`;
- `python_cpu_pytorch`.

The expected checks are:

- stored production C++ GPU prices and regenerated validation C++ GPU prices
  match to floating-point precision;
- regenerated C++ CPU Philox and C++ GPU Philox match to floating-point
  precision;
- regenerated Python CPU and Python GPU are statistically close;
- regenerated Python GPU and C++ GPU are statistically close.
- when path export exists, one stored production row is reconstructed through
  `tools/paths`, repriced from the exported paths, and compared with the stored
  production price.

This audit validates both the engine and the actual production pipeline:
parameters, curve inputs, row ids, seeds, simulation rule, output schema, and stored prices.
The path-reconstruction check closes the loop between the fused pricing kernel
and the path-only reconstruction kernel.

The production-to-validation pipeline is:

1. generate or choose the production model database;
2. generate or choose the production product database;
3. generate the production C++ GPU priced result database;
4. generate validation slices for the model and product databases;
5. reprice the validation model/product slice with C++ GPU, C++ CPU, PyTorch
   GPU, and PyTorch CPU;
6. run the matching validation notebook.

For early-exercise products priced by Longstaff-Schwartz, regress the realized
future cashflows discounted back to the current exercise date. Do not regress a
one-step value function as in Tsitsiklis-style dynamic programming. The
regression basis and exercise-date convention must be documented in the result
YAML.

For stochastic-volatility American options, matching prices across engines is
not sufficient: certification must also check that `exercise_policy.basis.state`
contains the complete simulated Markov state. Heston requires spot and variance;
the eight-factor rough Heston approximation requires spot and all eight factors.

The production-audit slice is a correctness audit, not a throughput benchmark.
For random product databases, the first 100 rows can contain many distinct
`step_count` values and therefore many small engine calls. That overhead is a
property of the audit slice, not of the production kernel. Performance claims
should use the production C++ GPU result timing, or a dedicated production
benchmark with enough rows to amortize grouping, launch, and allocation costs.
Both native CUDA and PyTorch GPU timings require a representative untimed
warm-up; otherwise the chart measures runtime initialization rather than the
pricing implementation.

Warm-up is mandatory, not optional. It must execute the actual pricing engine
with representative tensor dimensions, dtype, device, numerical branch, and
workspace requirements. A generic CUDA operation such as a random draw or an
exponential is insufficient because PyTorch loads many operators lazily.

Validation timing may record two distinct quantities:

- `wall_seconds` is one real pricing call after warm-up;
- `benchmark_seconds` is the representative hot-call wall time used by notebook
  timing tables and charts;
- `benchmark_repetitions` records the sample count;
- `benchmark_statistic` records the aggregation rule and is `median`;
- `benchmark_row_count` and `benchmark_workload` identify the measured batch;
- `warmup_calls` records representative untimed calls and is at least one.

### Timing Contract

All four engines in one notebook must measure the same logical pricing call.
The contract is mandatory:

1. Build Python dictionaries, aligned-row lists, native `ctypes` arrays, and
   packed host tensors before the timed backend call whenever those objects can
   be reused unchanged by every engine.
2. Run one representative untimed warm-up for every engine. It initializes
   CUDA contexts, PyTorch worker pools, OpenMP teams, reusable workspaces, and
   allocators outside the sample. It must use the same model, product, shape,
   numerical branches, and duplicated workload as the measured call.
3. Measure one complete hot call for substantial Monte Carlo validation
   workloads. Analytic fixed-income workloads, whose individual calls are too
   short to be stable, use their documented five-call median. Production
   generation always performs one call.
4. GPU wall time includes host-to-device transfer of prepared numerical input,
   device computation, synchronization, and device-to-host transfer of numerical
   output. It excludes registry parsing and final JSON/dictionary formatting.
5. CPU wall time covers the corresponding prepared numerical backend call. It
   excludes the same registry parsing and final formatting work.
6. CUDA `kernel_seconds` uses events around device work only. It is never
   compared directly with an end-to-end CPU or PyTorch wall time when making a
   throughput claim; it is shown to expose launch and transfer overhead.
7. A duplicated benchmark is one larger call, not many tiny calls averaged
   together. Its multiplier, row count, workload label, warm-up, and repetition
   count must be identical for all four engines in that notebook.
8. Timings from different workloads must not share one chart. In particular, a
   duplicated 1,000-row result cannot be displayed beside 100-row results.
9. Each result generator runs in a fresh process. Do not load all generators
   successively with `runpy` in one interpreter: PyTorch thread pools, CUDA
   allocators, library workspaces, and device clock state can contaminate later
   native CUDA measurements by more than an order of magnitude.

The automatic performance audit raises a warning when the CUDA kernel is not at
least twice as fast as PyTorch GPU wall time, when PyTorch GPU takes at least
half the PyTorch CPU time, or when C++ CPU is slower than PyTorch CPU. A warning
triggers profiling; it is not permission to change timing boundaries or inflate
the workload selectively. A documented algorithmic or launch-overhead exception
may remain only after the four boundaries and workloads have been verified.

Black-Scholes and Rough Bergomi are explicit exceptions to the CUDA
factor-of-two threshold. Their dynamics are highly efficient in native
vectorized PyTorch, so the required invariant is the engine hierarchy itself:
the custom CUDA kernel must beat PyTorch GPU on the same workload. Falling
below a 2x speedup is not an alert while that strict ordering remains valid;
PyTorch GPU beating CUDA remains an alert.

When a validation slice is too small to expose the expected performance
hierarchy, benchmark a single larger batch made by deterministically duplicating
the validation rows. The correctness outputs remain the original 100 rows. The
benchmark must record the duplicated row count, use a representative untimed
warm-up, and never mix a kernel-only CUDA time with full CPU or PyTorch
pipelines. Analytic fixed-income audits use
100,000 rows and 5 hot repetitions under this rule. European Monte Carlo rate
products use 5, and Bermudan Longstaff-Schwartz
products use 3. Repetitions apply only to validation audits. Production
generation always prices the database once after warm-up.

CPU may legitimately beat GPU for a 100-row analytic zero-coupon or swap
audit. Such a workload is too small to amortize GPU launch, synchronization,
and transfer costs. In that case, retain the 100 rows for correctness and use
the documented duplicated-batch throughput benchmark for performance. This is
a workload-size correction, not repetition designed to manufacture a result.

Even on the duplicated workload, closed-form CIR, Hull-White, CIR++, and G2++
zero-coupon batches are explicit launch-bound exceptions for the middle-engine
hierarchy.
Their C++ CUDA kernel must still beat PyTorch GPU; no ordering is imposed among
C++ CPU, PyTorch CPU, and PyTorch GPU for this elementary formula.

Shifted Black-76 caplets and European swaptions use the same launch-bound
exception. They are closed-form elementwise formulas: on the documented
100,000-row duplicated workload, the CUDA kernel must beat PyTorch GPU, while
no ordering is imposed among the three framework/CPU wall times.

## Four-Engine Reference Validation

For early development or debugging, it is still acceptable to create compact
manual validation grids. For a new model/product pair, create price-only
validation databases for:

- `python_cpu_pytorch`
- `python_gpu_pytorch`
- `cpp_cpu_philox`
- `cpp_gpu_philox`

When gradients are required, create the same four databases with a gradient
engine suffix, for example `delta_crn`.

The expected checks are:

- C++ CPU Philox and C++ GPU Philox match to floating-point precision.
- Python CPU and Python GPU are statistically close.
- Python GPU and C++ GPU are statistically close.
- Timings are coherent: C++ GPU is the production target; optimized C++ CPU
  with OpenMP and Python GPU form the intermediate tier; Python CPU is the
  slowest readability baseline. Python CPU materially outperforming C++ CPU is
  a failed performance sanity check and requires investigation.

Python uses native batched PyTorch RNG and is a benchmark/reference
implementation. It does not need pathwise reproducibility or row-by-row seeding.
It must still be good PyTorch code: vectorized, batched by compatible time-grid
shape, faithful to PyTorch tensor semantics, and written once with `device`
selecting CPU or CUDA. Avoid Python loops over rows or paths when tensorized
operations can express the same computation clearly.

C++ uses the project Philox implementation and is the reproducible pathwise
backend. C++ CPU code must follow the same modeling strategy from one model to
the next: validate inputs, precompute constants, use the shared Philox RNG
helpers, keep the inner path/time loop tight and explicit, and parallelize
independent result rows with the established OpenMP static batch wrapper. C++ CUDA kernels
must be production-quality kernels, with payoff accumulation fused into
simulation when this avoids unnecessary memory traffic. New models should look
familiar next to Heston and Rough Bergomi rather than introducing a private
style.

Manual validation examples should use the same experiment size unless the YAML
clearly documents a different purpose:

- full Cartesian product of the validation model and product databases;
- `target_dt: 1/52`;
- `num_paths: 16384`.

For the historical 10-row model grids and 50-row product grids, this means 500
priced rows per result database. New production-first audits should instead
slice the production databases.

## Validation Notebook Standard

Validation notebooks are grouped by model under
`notebooks/validation/<model_family>/`. Their filenames retain the model,
product, output, audit purpose, and short database version.

Every validation notebook is generated by
`tools/notebooks/build_validation_audits.py`. "Same format" means the same
section order, section names, comparison table, timing rows, timing chart, and
regeneration-command layout. Model-specific wording and path reconstruction
code may change; presentation and validation coverage may not.

Every production-audit price-only consistency table must contain exactly these
four rows, in this order:

1. stored production C++ GPU versus regenerated C++ GPU;
2. C++ CPU Philox versus C++ GPU Philox;
3. Python CPU PyTorch versus Python GPU PyTorch;
4. C++ GPU Philox versus Python GPU PyTorch.

Manual validation notebooks that do not have a production result source may
omit the first row and keep the last three rows.

Do not add C++ CPU versus Python CPU, C++ CPU versus Python GPU, or any other
comparison to the certification notebook. Additional diagnostics belong in
tests or dedicated debugging scripts.

For a price-and-gradient audit, repeat the canonical four comparisons for each
requested output, starting with `price` and then each gradient in the order
declared by the result YAML:

1. stored production C++ GPU versus regenerated C++ GPU;
2. C++ CPU Philox versus C++ GPU Philox;
3. Python CPU PyTorch versus Python GPU PyTorch;
4. C++ GPU Philox versus Python GPU PyTorch.

After those rows, add exactly two price-reuse checks: price-only versus
price-and-gradient for C++ GPU first, then C++ CPU. For one gradient this gives
ten rows: four price comparisons, four gradient comparisons, and two reuse
comparisons. `tools.validation.audit.coherence_frame` owns this ordering.

Every timing table must contain exactly the same four engine rows:

1. `cpp_gpu_philox`;
2. `python_gpu_pytorch`;
3. `cpp_cpu_philox`;
4. `python_cpu_pytorch`.

The visible timing table contains only `wall seconds` and `kernel seconds`.
`wall seconds` is the hot end-to-end call for every engine; when a duplicated
benchmark exists, it is `benchmark_seconds`. `kernel seconds` is populated only
for C++ CUDA. Simulation, payoff, transfer, repetition, and throughput details
may remain in result metadata but are not displayed in validation notebooks.
The timing chart always plots `wall seconds`, where lower is better.

The two CUDA measures have fixed semantics. `wall seconds` surrounds the whole
hot pricing call after warm-up, including row grouping, host preparation,
transfers, launches, synchronization, and output collection. `kernel seconds`
uses CUDA events around device computation only. Their ratio is therefore not a
target: it approaches one for large device-bound batches and can be much larger
for tiny analytic kernels or highly fragmented `step_count` groups. Since the
clocks differ, accumulated event measurements may exceed the host clock by a
small resolution error; persisted timings clamp `wall seconds` to at least
`kernel seconds`.

For deterministic analytic throughput audits, inputs are materialized before
the timed backend call and result dictionaries are materialized afterwards.
This matches the native C++ analytic timing boundary and prevents Python list
construction from masking the CPU/GPU backend comparison. The stored prices
still include the complete conversion path; only the benchmark boundary is
restricted to prepared backend execution.

Timing rows and chart bars must always use this fixed order, independently of
the measured values: C++ CUDA, PyTorch GPU, C++ CPU, PyTorch CPU. Do not sort
them by wall time. This stable presentation makes notebooks directly
comparable and reflects the natural target hierarchy without hiding a timing
regression.

Timing in validation notebooks is operational context. It should be compact and
must not turn the notebook into a performance explanation. Detailed analysis of
`step_count` fragmentation, cold-start overhead, launch counts, or throughput
belongs in documentation, profiling scripts, or production benchmark reports.

The required production-audit notebook sections are exactly:

- `Load Data And Timings`;
- `Dataset Coherence`;
- `Path Reconstruction Check` when reproducible path export exists;
- `Regeneration Commands`.

When C++ path export is available, keep a path-reconstruction repricing check in
the production-audit notebook or in a dedicated path audit.

The notebook is a certification report, not an exploratory scratchpad.
Product-specific diagnostics do not belong there. Economic autocall
diagnostics, barrier parity, exercise-policy checks, and similar invariants are
automated tests or future production-analysis notebook content.

For discretely monitored barrier calls, paired knock-in and knock-out product
databases must share the same strike, maturity, barrier, observation schedule,
and result seed row by row. Automated tests enforce the pathwise partition
`knock-in + knock-out = vanilla call` and equality of the up/down vanilla sums
under C++ Philox.

Relative price errors are reported only when
`max(abs(left_price), abs(right_price)) >= 1e-4`. With normalized spot or
nominal equal to one, this excludes prices below one basis point, for which a
relative percentage is numerically unstable and economically uninformative.
Absolute errors and z-scores still use every row. The table reports the number
of rows retained for relative-error metrics.

## New Model/Product Deliverable

Reading this document must be sufficient to implement a new pair consistently.
For example, a request for a barrier option under a new model implies all of the
following deliverables without redefining the protocol:

- one optimized, batched PyTorch implementation shared by CPU and GPU through
  the `device` argument;
- one optimized C++ CPU implementation following the established validation,
  precomputation, Philox, and tight-loop structure;
- one production-quality fused C++ CUDA implementation using the shared Philox
  mapping and deterministic reductions;
- one production C++ GPU result database, initially allowed to be small;
- production model/product/result validation slices;
- four price-only repriced validation audit result databases;
- when gradients are requested, four additional price-and-gradient databases;
- C++ CPU/GPU floating-point reproducibility tests;
- Python CPU/GPU and C++ GPU/Python GPU statistical-coherence tests;
- price-only versus price-and-gradient C++ path-reuse tests;
- path reconstruction and repricing when the model exposes path export;
- a notebook generated by `tools/notebooks/build_validation_audits.py`, with no
  product-specific certification rows.

Optimization is part of the deliverable. Python must use idiomatic vectorized
and batched PyTorch rather than row/path loops. C++ CPU and CUDA code must follow
the same architecture and naming philosophy as existing validated models, so a
reader can move between implementations without learning a new private design.

## Result Requirements

A result YAML must include `title`, `format`, `database_id`, `json_path`,
`generation_script`, `summary`, `time_grid`, `outputs`, `model_database`,
`product_database`, `result_construction`, and `timing`. A curve-dependent
result additionally includes `curve_database`. Database references contain
both `id` and `json_path`; the same nested references are stored in result JSON.

The required nested metadata includes:

- `summary.row_count` and `summary.source_files`;
- `time_grid`, either with numerical-grid details, an exact-transition rule, or
  an explicit analytic-not-applicable rule;
- `outputs`;
- `model_database.id` and `model_database.json_path`;
- `curve_database.id` and `curve_database.json_path` when applicable;
- `product_database.id` and `product_database.json_path`;
- `result_construction.rule`;
- `timing.wall_seconds`.

`summary.source_files` contains the primary model/product implementation only:
the specialized `.cu`, `.cpp`, or PyTorch `.py` selected by the engine. Common
headers, the C API, and tools are dependencies rather than result source files
and are not repeated in YAML.

Result JSON rows should stay compact:

```text
id, model_id, [curve_id], product_id, [seed], outputs
```

Result JSON top-level metadata contains the same nested database references and
`result_construction` as YAML. Flat database-id fields are not permitted.

Do not store full trajectories in result databases.

## Time Grid

The active convention is:

```yaml
time_grid:
  rule: nearest integer step count to target dt
  target_dt: 1/52
  step_count: round(maturity / target_dt)
  effective_dt: maturity / step_count
```

The JSON does not store per-row `num_steps`; it is reconstructed from product
maturity and result YAML.

Execution code may group rows by exact computed `step_count` and chunk large
groups for memory. This must not change row ids, seeds, or output order.

Bucketizing or capping step counts changes the numerical grid and must be
declared as a new `time_grid` rule.

## Path Reconstruction

When path reconstruction is part of a validation case, provide model-level path
helpers under `tools/paths`. They should regenerate the exact same spot
trajectories from row id, seed, model row, product maturity, and time-grid rule.
They should also expose a direct simulation entry point for generating paths
from model parameters, maturity, seed, and grid settings without a result
database.

Validation should reprice at least one row from reconstructed paths and compare
against the stored result.

Native CPU/CUDA comparisons use `1e-12` by default. A narrowly documented
`1e-10` tolerance is acceptable for rough-Heston accumulated path statistics
such as maxima and realized volatility because CPU sequential sums and CUDA
tree reductions do not share an addition order. American-option exercise
boundaries use `1e-4`; tiny path differences can change a regression decision.
These exceptions remain far below Monte Carlo uncertainty and must not be
silently generalized to unrelated products.

## Scaling Production

Once the production slice audit passes, scaling should use the same C++ CUDA
engine, same YAML numerical protocol, and same row/seed mapping. Large random
model or product parameter databases belong under `registry/production`. Their
YAML must document the sampling law, bounds, RNG, seed, and row count.

For million-row product databases:

1. assign stable row ids and seeds;
2. compute `step_count` from the result YAML rule;
3. group by exact `step_count`;
4. chunk groups for memory;
5. write outputs in deterministic row order.
